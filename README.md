# ScratchBird Benchmarks

Benchmark harness for FirebirdSQL, MySQL, and PostgreSQL with comparable outputs for:

- Native baseline validation
- Cross-engine performance and behavior comparison
- ScratchBird side-by-side analysis against native engines

This repo is centered on `scripts/run-benchmark-matrix.sh`, which runs a repeatable matrix and writes both detailed artifacts and a unified comparison CSV.

## What You Need

- Docker Engine/Desktop with daemon access (`docker info` must work)
- Python 3.8+ (`python3`)
- Optional but recommended: virtualenv and `pip install -r requirements.txt`
- Local engine source clones for regression suite mounting:
  - Firebird test repo path (`FIREBIRD_REPO_PATH`)
  - MySQL source clone (`MYSQL_REPO_PATH`)
  - PostgreSQL source clone (`POSTGRESQL_REPO_PATH`)

Copy and edit environment defaults:

```bash
cp .env.example .env
```

## Quick Start

```bash
cd /home/dcalford/CliWork/ScratchBird-Benchmarks

# Optional Python environment
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

# Recommended full native baseline run
SCRATCHBIRD_PG_QUERY_TIMEOUT_MS=30000 \
./scripts/run-benchmark-matrix.sh \
  --engines=firebird,mysql,postgresql \
  --suites=regression,stress,acid,performance,tpc-c,tpc-h,engine-differential \
  --report --compare
```

The matrix writes to `results/matrix-YYYYMMDD-HHMMSS/` by default.  
To force an output directory:

```bash
BENCHMARK_MATRIX_OUTPUT=results/matrix-my-run \
./scripts/run-benchmark-matrix.sh --engines=firebird,mysql,postgresql --suites=regression,stress,acid,performance,tpc-c,tpc-h,engine-differential --report --compare
```

## Core Commands

- Full matrix (recommended): `./scripts/run-benchmark-matrix.sh ...`
- Single engine + suite: `./scripts/run-benchmark.sh <engine> <suite> [--report]`
- Start/stop engine manually: `./scripts/start-engine.sh <engine> <start|stop|status|logs>`
- Regenerate unified CSV from existing run:
  - `python3 scripts/generate-unified-comparison-csv.py --output-root <matrix-output-root>`

## Output Layout

For a matrix output root like `results/matrix-20260316-full-proceed`:

- `matrix-summary.json`
  - Machine-readable run summary and suite execution metadata.
- `.matrix-runs.tsv`
  - One line per `(engine, suite)` execution with status and duration.
- `matrix-comparison-unified.csv`
  - Consolidated cross-engine comparison table (primary decision file).
- `comparison-<suite>/benchmark_comparison_*.txt`
  - Suite-level cross-engine text comparison report (when `--compare` is used).
- `<engine>/<suite>/*.json`
  - Raw suite result JSON for each engine.
- `<engine>/<suite>/reports/benchmark_comparison_*.txt`
  - Per-engine suite report text (when `--report` is used).

## How To Read Reports

### 1) `matrix-summary.json`

Use this for run integrity and completion checks:

- `total_suite_runs` should match expected matrix size.
- `failed_suite_runs` and `result` indicate overall health.
- `suite_runs[]` includes per-run:
  - `engine`
  - `suite`
  - `status`
  - `duration_seconds`
  - `exit_code`
  - `output_dir`

### 2) `benchmark_comparison_*.txt` reports

Human-readable comparison summaries generated by `system-info/submit/result_formatter.py`.

- Use for quick narrative review and sharing.
- Use JSON/CSV artifacts for strict automated analysis.

### 3) `matrix-comparison-unified.csv` (Consolidated Report)

This is the canonical cross-engine comparison table.

Schema:

- `run_id`
- `suite`
- `metric`
- one column per engine (`firebird`, `mysql`, `postgresql`)

Metric naming patterns:

- `matrix.*` -> execution metadata from matrix runner
  - `matrix.status`
  - `matrix.exit_code`
  - `matrix.duration_seconds`
- `totals.*` -> regression aggregate metrics
- `summary.*` -> suite summary metrics from result JSON
- `results.*` / `test_results.*` -> counts derived from suite result payloads
- `artifact.result_json` -> relative path to source JSON artifact

## Using The Consolidated CSV For Decisions

Example: inspect stress runtime and pass/error patterns quickly:

```bash
csv=results/matrix-20260316-full-proceed/matrix-comparison-unified.csv
rg ',stress,matrix.duration_seconds,' "$csv"
rg ',stress,summary.passed,' "$csv"
rg ',stress,summary.errors,' "$csv"
```

Example: inspect ACID outcome deltas:

```bash
csv=results/matrix-20260316-full-proceed/matrix-comparison-unified.csv
rg ',acid,summary.passed,' "$csv"
rg ',acid,summary.errors,' "$csv"
```

Example: inspect engine-differential execution time:

```bash
csv=results/matrix-20260316-full-proceed/matrix-comparison-unified.csv
rg ',engine-differential,matrix.duration_seconds,' "$csv"
```

For automation, parse `matrix-comparison-unified.csv` with Python, pandas, or your CI tooling and apply thresholds per metric.

## Recommended ScratchBird Comparison Workflow

1. Run native matrix baseline and archive output root.
2. Run equivalent ScratchBird-targeted runs in each dialect mode.
3. Generate consolidated CSVs for each run set.
4. Compare:
   - Matrix execution health (`matrix.status`, `exit_code`)
   - Pass/fail and error signatures (`summary.*`, `totals.*`)
   - Relative performance (`matrix.duration_seconds`, suite-specific summary timing metrics)
5. Track regressions by metric over time in CI.
